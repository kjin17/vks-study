VKS란 무엇인가 — vSphere Kubernetes Service

인프라 전체를 “쿠버네티스화”

VKS는 기존 vSphere 가상화 인프라 전체를 Kubernetes 개념으로 추상화하여 kubectl로 관리합니다.

vSphere Namespace↔Namespace Resource Pool 성격 — 리소스·정책 경계
Datastore↔StorageClass
PersistentVolume (PV)↔VMDK First Class Disk (FCD)
CNI↔vSphere Networks NSX / VDS

Supervisor란?

vSphere 클러스터에 Kubernetes 워크로드 관리 기능을 활성화한 관리용 컨트롤 플레인
vSphere 클러스터(ESXi 호스트 그룹)에 Kubernetes 워크로드 관리 기능을 활성화한 관리용 Kubernetes 컨트롤 플레인입니다.
역할
  • vCenter가 관리하는 관리 평면
  • vSphere Namespace 단위로 리소스·권한(RBAC)·정책 부여
배포 주체
  • 앞의 3가지(VM Service · vSphere Pod · VKS Cluster)를 배포하는 주체
  • 플랫폼 확장은 Supervisor Services로 추가

Supervisor Services란?

Supervisor에 설치해 플랫폼 기능을 추가하는 확장 서비스 (오퍼레이터/컨트롤러)
Supervisor에 설치하는 확장 서비스(Operator / Controller) — 플랫폼 기능을 추가합니다.

  • Harbor — 컨테이너 레지스트리(Registry)
  • Contour — Ingress
  • Argo CD · cert-manager 등

Component

VKS Service란?

Supervisor Services 중 하나 — 워크로드 K8s 클러스터의 프로비저닝·라이프사이클 담당
Supervisor Services 중 하나로, VKS(워크로드 K8s) 클러스터의 프로비저닝과 라이프사이클을 담당하는 서비스입니다.
동작 방식
  • Cluster API (CAPI) + CAPV를 통해 클러스터 관리
  • builtin-generic ClusterClass 기반으로 클러스터 정의
  • 생성 · 업그레이드 · 스케일을 선언적으로 수행
버전 예시
  • VKS 3.6 / 3.7 (VCF 9.1)
  • Supervisor Service로 설치·활성화 후 클러스터 생성 가능

vSphere Kubernetes Release (VKr)란?

VKS 클러스터 노드를 만드는 배포 단위 — 서명된(signed) 노드 이미지/OVA
VKS 클러스터 노드를 만드는 배포 단위(서명된 노드 이미지 / OVA)입니다.
한 묶음으로 검증·배포
  • 특정 Kubernetes 버전
  • 노드 OS (Photon / Ubuntu / RHEL)
  • 핵심 컴포넌트를 한 묶음으로 검증
사용
  • 클러스터는 VKr을 지정해 노드를 프로비저닝
    예: VKr v1.36
  • 업그레이드 시 VKr 버전을 올림

Networking

서비스 노출 방식 비교 — Upstream K8s vs VKS

기존 NodePort / 단일 Ingress 구조에서 NSX-ALB 기반 전용 LoadBalancer · 클러스터 고유 Ingress로 진화
Upstream K8s 기존 환경
① NodePort
노드 IP + 고정 포트(30000‑32767)로 직접 노출
→ 외부 LB 장비를 별도 구성·관리해야 함
② 클러스터 공용 Ingress Controller
NGINX 등 1개를 클러스터 공용으로 공유, HTTP/S 라우팅
→ VIP 충돌 위험 · 팀별 정책 격리 불가
⚠ 인프라 팀이 외부 장비를 수동 관리해야 하며, 클러스터 간 LB 공유로 단일 장애점 발생 가능
VKS + NSX-ALB VCF 9.1
① LoadBalancer 타입 Service
NSX Advanced Load Balancer(AVI)가 자동으로 전용 VIP 할당
→ type: LoadBalancer 선언만으로 외부 IP 즉시 발급
② 클러스터 고유 Ingress Controller
VKS 클러스터마다 독립 Ingress Controller 배포 가능
→ 팀·서비스별 정책 격리, VIP 충돌 없음
✓ 인프라 수동 개입 없이 K8s 선언만으로 L4/L7 서비스 완결

Migration Notes

마이그레이션 유의점

업그레이드 전 반드시 확인 — 잘못 선택 시 이미지 해석/업그레이드 실패
  • TKC API 지원 완전 제거 — TanzuKubernetesCluster API는 3.7에서 제거(VKr 1.32가 마지막). VKS Cluster(ClusterClass) API로 전환 필요.
  • 업그레이드 전 VKr 하한 충족 — 3.6은 1.32+, 3.7은 1.33+ 부터 업그레이드 경로 지원.
  • ClusterClass 리베이스(rebase) — 새 K8s 업그레이드 전 ClusterClass를 v3.6 / v3.7로 먼저 리베이스.
  • 에어갭 vs 인터넷 연결 YAML 구분 — 배포 YAML을 환경에 맞게 선택. 잘못 선택 시 이미지 해석(image resolution) 실패.
