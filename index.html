01

VKS란?

vSphere Supervisor 위에서 Kubernetes 클러스터의 프로비저닝과 라이프사이클을 관리하는 서비스입니다. (구 Tanzu Kubernetes Grid Service)

핵심 계층 구조

  • vSphere / ESXi — 컴퓨트·스토리지·네트워크 인프라 기반
  • Supervisor — 관리용 Kubernetes 컨트롤 플레인 (management K8s control plane)
  • vSphere Namespace (ns) — 테넌시·리소스 쿼터·정책 경계
  • VKS 워크로드 클러스터 — 실제 애플리케이션이 도는 K8s 클러스터. 버전 단위는 VKr = vSphere Kubernetes Release
기반 기술 · Foundation

Cluster API (CAPI) + CAPV (Cluster API Provider vSphere) 기반으로 동작하며, builtin-generic ClusterClass로 클러스터 토폴로지를 선언적으로 정의합니다.

9.1에서 달라진 운영 체계

  • 관리 도구 통합 — 기존 kubectl-vsphere 플러그인 → VCF CLI vcf + VCF context 로 통합
    • 클러스터 컨텍스트 접속·조회를 vcf 명령 체계로 일원화
  • 이미지 / 패키지 공급 — VCF 9.1 내장 Software Depot (OCI 레지스트리)
    • 에어갭 (air-gapped) 환경에서도 외부 레지스트리 없이 구성 가능
  • 애드온 (Add-ons) — 큐레이션된 VKS Standard Packages를 vcf addon 으로 설치
    • cert-manager, Prometheus 등 검증된 패키지 제공
    • VKr 1.35부터 CNI(Antrea / Calico / Cilium)도 애드온 프레임워크 (AddonConfig)로 전환
# VCF CLI 컨텍스트 접속 및 애드온 설치 예시
vcf context use <supervisor-context>
vcf cluster list --namespace <ns>
vcf addon install cert-manager --cluster <vks-cluster>

04

9.1 신규 기능 하이라이트

VCF 9.1 / VKS 3.6·3.7에서 추가된 복원력·보안·운영 기능입니다.

1

5노드 컨트롤 플레인

컨트롤 플레인을 최대 5노드로 확장해 극한 복원력 확보.

VKr 1.36.1+ / 1.35.5+ / 1.34.8+
2

인플레이스 노드 업데이트 (In-place)

롤링 재생성 없이 노드 구성 변경을 제자리에서 적용.

3

kube-proxy 비활성화 API

Cilium 등 자체 프록시를 갖춘 CNI 대상, kube-proxy를 끌 수 있는 API.

VKr 1.35+
4

TLS 프로파일 구성

minimumTLSProtocol / tlsCipherSuites 를 apiserver · etcd · kubelet 에 일괄 적용.

5

커스텀 RuntimeClass

ContainerD ulimit 등 조정으로 DB 등 고부하 워크로드 대응.

6

etcd 튜닝

최대 요청 크기 조정 — 대용량 GitOps 트랜잭션 대응.

7

보조 NIC vfio-pci / SR-IOV

Telco 5G CNF, DPDK 가속을 위한 보조 NIC 바인딩 지원.

8

CIDR 중복 방지 검증 강화

Supervisor IP 대역과 Pod / Service CIDR 충돌을 사전 차단.

9

/usr 읽기 전용 마운트

Photon 노드에서 DaemonSet 변조 방어를 위한 읽기 전용 /usr.

10

FIPS 기본 번들

FIPS 검증 번들 기본 제공 — Canonical Pro 구독 불필요.

05

마이그레이션 / 업그레이드 유의점

  1. TKC API 클러스터 지원 완전 제거 (3.7)

    TanzuKubernetesCluster (TKC) API 기반 클러스터는 VKS 3.7에서 지원이 완전히 제거됩니다. VKr 1.32가 TKC를 지원하는 마지막 릴리스이므로, TKC 의존 클러스터는 사전 전환이 필요합니다.

  2. 업그레이드 전 VKr 하한 확인

    VKS 3.6은 VKr 1.32 이상, VKS 3.7은 VKr 1.33 이상이 필요합니다. 하한 미만 클러스터는 중간 버전을 거쳐 단계적으로 올려야 합니다.

  3. ClusterClass 리베이스 필수

    새 Kubernetes 버전으로 업그레이드하기 전에 builtin-generic-v3.6.0 / v3.7.0 으로의 ClusterClass 리베이스가 선행되어야 합니다.

  4. 에어갭 vs 인터넷 연결 — 서비스 YAML 선택

    에어갭 vCenter 9.1은 "legacy" 서비스 YAML(공개 레지스트리 참조)을, 인터넷 연결 vCenter 9.1은 Software Depot 참조 YAML을 사용합니다. 잘못 선택하면 이미지 해석에 실패합니다.

주의 · Caution

업그레이드 순서는 ① TKC 클러스터 전환 → ② VKr 하한 충족 → ③ ClusterClass 리베이스 → ④ 환경에 맞는 서비스 YAML 확인 순으로 검증하세요. 특히 에어갭 환경의 YAML 선택 오류는 배포 시점에야 이미지 실패로 드러나므로 사전 점검이 중요합니다.

06

마무리

One-line Summary

VCF 9.1의 VKS는 내장 Software Depot + VCF CLI 통합 + Kubernetes 1.36 + 엔터프라이즈 복원력·보안(5노드 CP · OIDC · TLS 프로파일 · FIPS)으로 프로덕션 운영 성숙도를 크게 끌어올린 릴리스입니다.
