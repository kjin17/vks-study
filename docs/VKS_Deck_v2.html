BROADCOM 1 · VKS 아키텍처 개요

VKS 아키텍처 개요

VKS = vSphere Kubernetes Service
구(舊) TKGs(Tanzu Kubernetes Grid Service)
VCF 9.1부터 VKS로 공식 명칭 통합

핵심 개념: vSphere 인프라 위에서 동작하는 완전 관리형 Kubernetes 플랫폼

  • Supervisor: K8s API Server 내장 (vCenter 통합)
  • ESXi 호스트가 K8s Worker Node 역할
  • Control Plane: 5노드 HA 9.1 NEW
  • Worker: PhotonOS 5 / Ubuntu 22.04
  • NSX 또는 vDS 기반 네트워킹

BROADCOM 3 · Supervisor 3가지 워크로드 유형

Supervisor가 배포할 수 있는 3가지 워크로드

🖥️ VM Services
  • 기존 VM을 K8s API로 관리
  • VM Operator CRD 사용
    VirtualMachine
  • 컨테이너화 없이 K8s 관리
  • VM 생명주기 자동화
  • Windows VM 지원
💼 레거시 앱 · DB · Windows
🫧 vPod (가상화된 Pod)
  • ESXi 위에서 직접 실행되는 Pod
  • VM 격리 수준 보안
  • 별도 Worker Node VM 불필요
  • Supervisor 네임스페이스 직접 실행
  • 빠른 프로비저닝
🔒 보안 민감 · 소규모 서비스
☸️ VKS Cluster
  • 완전한 K8s 클러스터 프로비저닝
  • CP + Worker Node 모두 VM
  • 표준 K8s API 완전 호환
  • 멀티 테넌트 독립 클러스터
  • Helm, kubectl 모두 사용
🚀 마이크로서비스 · CI/CD · AI/ML
📌 세 가지 유형은 동일 Supervisor 네임스페이스에서 혼용 가능 — vSphere 리소스(CPU·메모리·스토리지)는 Namespace 할당량 내에서 공유

BROADCOM 8 · 마이그레이션 유의사항

9.0 → 9.1 업그레이드 체크리스트

  • TKC → VKS API 마이그레이션
    자동화 스크립트 활용 가능
  • Control Plane 3→5노드 확장
    추가 VM 리소스 확보 필요
  • PhotonOS 4 → PhotonOS 5
    노드 이미지 신규 다운로드
  • imageBaker 설정
    에어갭 환경에서 필수
  • VCF CLI 설치
    기존 kubectl 워크플로 업데이트
  • OIDC 통합 확인
    기존 외부 IdP 설정 검토
  • NSX-T → NSX/VDS 마이그레이션
    네트워크 구성 변경 시 다운타임 고려
업그레이드 순서
1️⃣ vCenter / Supervisor 업그레이드
↓
2️⃣ Control Plane 노드 업그레이드
↓
3️⃣ Worker Node 롤링 업그레이드
↓
4️⃣ TKC → VKS API 마이그레이션
⏱️ 롤링 업그레이드로 서비스 중단 최소화
Worker Node는 1개씩 순차 교체
