01

왜 VKS인가 — 고객의 고민과 해결

VKS는 이미 운영 중인 vSphere 인프라를 그대로 쿠버네티스 플랫폼으로 전환하여, 전통적인 VM과 최신 컨테이너 애플리케이션을 하나의 검증된 플랫폼에서 운영하게 합니다. 별도의 쿠버네티스 스택을 새로 구축·운영할 필요가 없습니다.

기존 고민 Pain PointVKS의 해결
컨테이너용 K8s를 별도 구축·운영 사일로 · 이중 투자vSphere 위에서 K8s를 네이티브로 제공 — 기존 인프라 재사용
개발팀 셀프서비스 요구 vs 인프라팀 통제개발자는 표준 K8s API(YAML), 관리자는 vCenter 거버넌스로 양립
K8s 버전·애드온 파편화, 보안 패치 부담검증된 VKr 릴리스 번들 + 24개월 지원 라이프사이클
규제·에어갭 환경의 컨테이너 도입 난이도VCF 9.1 내장 Software Depot 기반 에어갭 배포
클라우드 대비 비용·거버넌스 가시성 부족FinOps 비용 분석 + 지속 컴플라이언스 (VCF 9.1)
한 줄 메시지

"가상화 위에 쿠버네티스를 얹는 것"이 아니라, 가상화 인프라 자체를 쿠버네티스로 표현·제어합니다.

차별점

이런 고객에게 Use Cases

  • 애플리케이션 현대화 — 레거시 VM과 신규 컨테이너를 단일 플랫폼에서 점진 전환
  • AI/ML 플랫폼 — Custom Zone + 지능형 노드 풀 배치로 GPU 워크로드 격리·최적 배치
  • Telco / 5G — SR-IOV·DPDK(vfio-pci) 보조 NIC로 CNF 고성능 네트워킹
  • 멀티테넌시 / CSP — 테넌트별 EncryptionClass 키, 다중 네트워크, Supervisor당 500 클러스터
  • 규제 산업 / 에어갭 — 인터넷 차단 환경에서도 안전한 컨테이너 배포

04

VKS 아키텍처

VKS는 vSphere 환경에 내장된 쿠버네티스 컨트롤 플레인입니다. 아래는 인프라부터 워크로드 클러스터까지의 계층 구조입니다.

Workload · 개발팀이 사용
VKS 워크로드 클러스터 (Guest / TKG Cluster)
실제 앱이 도는 업스트림 K8s 클러스터 · 버전 단위 = VKr (vSphere Kubernetes Release)
VM Services전통 VM을 K8s API로 관리
vPodESXi 위 직접 실행 네이티브 Pod
▲   프로비저닝 · 라이프사이클   ▲
Control Plane
Supervisor (관리용 K8s 컨트롤 플레인)
vCenter 내 프로비저닝 · vSphere Namespace로 테넌시·쿼터·정책 경계 제공 · CAPI/CAPV + ClusterClass 기반
▲
Infrastructure
vSphere / ESXi
컴퓨트·스토리지·네트워크 기반 · 각 ESXi가 Spherelet(Kubelet의 vSphere 구현)으로 워커처럼 동작
기반 기술

Cluster API (CAPI) + CAPV(Cluster API Provider vSphere) 기반으로 동작하며, builtin-generic ClusterClass로 클러스터 토폴로지를 선언적으로 정의합니다.

06

Supervisor가 배포하는 3가지 워크로드

1

VM Services

전통 VM을 K8s API(YAML)로 관리. VM Operator CRD(VirtualMachine)로 생명주기 자동화.

기존 VM → K8s 관리
2

vPod

ESXi 하이퍼바이저 위에서 직접 실행되는 네이티브 Pod. VM급 격리 보안 + 별도 Worker VM 불필요.

하이퍼바이저 네이티브
3

VKS Cluster

독립적 업스트림 K8s 클러스터. CP+Worker 모두 VM, 표준 K8s API 완전 호환, 멀티테넌트.

Helm · kubectl
07

Supervisor Service · VKS Service · VKR

A

Supervisor Service

vSphere 위 K8s 컨트롤 플레인 서비스. 개발자에 K8s API 엔드포인트 제공, IT 관리자는 vCenter에서 보안·수명주기·정책을 중앙 집중 관리.

B

VKS Service

게스트 클러스터의 생성·스케일·업그레이드 전체 수명주기 자동화. CNCF 인증 표준 K8s를 온디맨드로 제공, 퍼블릭 클라우드급 셀프서비스.

VKR / TKr — vSphere Kubernetes Release

Broadcom이 테스트·호환성 검증을 완료한 K8s 바이너리 + OS 이미지 + 필수 애드온(CoreDNS·CNI·CSI 등)이 하나로 패키징된 릴리스. 최신 K8s를 안정적으로 배포하고 클러스터 간 버전·구성 파편화를 방지합니다. 지원은 24개월 라이프사이클에 귀속됩니다.

10

VKS Add-ons 생태계

클러스터 기능을 확장하는 검증된 컴포넌트 모음. 3.7.0부터 "Core/Standard 패키지" 용어를 폐기하고 통합 VKS Add-ons 프레임워크로 전환했습니다. 호환성은 Supervisor가 아니라 클러스터의 K8s(VKr) 버전 기준입니다.

티어소유/검증Broadcom 지원 범위
ProductBroadcom 엔지니어링 큐레이션·검증배포 + 라이프사이클 + 런타임 트러블슈팅까지 완전 지원
PartnerISV(독립 SW 벤더) 검증배포·라이프사이클 지원, 런타임은 파트너 에스컬레이션
EcosystemBroadcom 큐레이션 OSS패키징·서명·라이프사이클만, 런타임은 업스트림
Community미검증/사용자 빌드 OSS공식 지원 없음
운영 팁

Cilium 고급 기능(L7 프록시·Ingress·Egress GW·Gateway API·eBPF 마스커레이딩)은 kube-proxy 비활성화가 전제입니다 — VKS 3.7의 kube-proxy 비활성화 API와 세트로 사용하세요. (현재 Istio 애드온과 Cilium CNI는 비호환)

11

업그레이드 · 운영 체크리스트

  1. 업그레이드 경로 확인

    Supervisor의 VKS 버전과 모든 클러스터 VKr을 먼저 상향 (3.7은 Supervisor VKS 3.4+, 클러스터 VKr 1.33+). TKC API 클러스터는 3.7 전에 마이그레이션 필수.

  2. ClusterClass 리베이스

    builtin-generic-v3.5.0 → v3.6.0 → v3.7.0 순차 리베이스. 구버전은 deprecated 후 제거 예정.

  3. 에어갭 / YAML 선택

    3.7은 인터넷 연결 vCenter 9.1+는 일반판, 9.0 이하·에어갭은 -legacy YAML (잘못 고르면 이미지 레지스트리 해석 실패).

  4. 애드온 주의

    Gatekeeper 사용 시 업그레이드 전 웹훅 비활성화(KB 433183) · AKO 1.13.4→2.x 전 기본 StorageClass 확인 · 패키지→애드온 마이그레이션 시 values 파일 -f 지정.

  5. 최신 포인트 릴리스 사용

    3.5.1 / 3.6.3 등 누적 버그 수정본 권장.
